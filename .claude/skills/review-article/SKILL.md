# ブログ記事レビュー

執筆済み記事を多角的にレビューし、改善点を提示する。

## 事前確認（必須）

レビュー前に以下を必ず読むこと：

- `guidelines/writing-style.md` — 文体・人格の基準
- `guidelines/article-templates.md` — 構成テンプレートの基準
- `guidelines/article-categories.md` — カテゴリ別の要件

## 対象ファイルの特定

- ユーザーがファイルを指定した場合はそのファイルを読む
- 指定がない場合は `completed/` 配下のファイルを一覧表示してユーザーに選ばせる

## レビュー観点

### 1. 文体チェック

- [ ] 一人称が「自分」になっているか（「私」「僕」は NG）
- [ ] です・ます調で統一されているか
- [ ] 断定的な表現を避け「個人的には〜」「〜と感じています」などの謙虚な表現を使っているか
- [ ] 「〜していきます」「〜していきたいと思います」などの進行形が使われているか
- [ ] 絵文字が技術・組織記事に含まれていないか
- [ ] 1文が長すぎないか（目安: 100文字以内）
- [ ] 「〜すべき」「〜ねばならない」などの強制的な表現を使っていないか

### 2. 構成チェック

- [ ] フロントマター（title / published / updated / url / tags）が揃っているか
- [ ] 「こんにちは、プレックスの[石塚](https://twitter.com/ij_spitz)です。」で始まっているか
- [ ] 導入（背景・動機）が2〜3段落あるか
- [ ] `[:contents]` が導入の直後にあるか
- [ ] 「おわりに」または「終わりに」セクションがあるか
- [ ] 技術・組織記事に採用情報リンクがあるか `[https://dev.plex.co.jp/:embed:cite]`

### 3. 内容の充実度

- [ ] 記事の背景・動機が明確に説明されているか
- [ ] 具体例・数字・データが含まれているか
- [ ] 失敗談や課題が正直に書かれているか（美化していないか）
- [ ] 技術記事にコード例があるか（実装前後の比較があると尚良）
- [ ] 抽象論だけでなく再現可能な具体的情報があるか
- [ ] アウトラインの全項目が網羅されているか（`outlines/` と照合）

### 4. はてな記法チェック

- [ ] 外部リンクは `[https://example.com/:embed:cite]` 形式になっているか
- [ ] 画像は `[f:id:username:imageid]` 形式になっているか
- [ ] コードブロックに言語名が指定されているか（` ```ruby ` など）
- [ ] 目次が `[:contents]` で記述されているか

### 5. リンクの有効性チェック

記事内のリンクを WebFetch で検証する。

- 外部リンクが存在するURL（`http://` または `https://` で始まる）をすべて抽出する
- 各URLに対して WebFetch でアクセスし、404や接続エラーがないか確認する
- 問題があるリンクは代替URLを提案する

## レビュー結果の出力形式

レビュー結果は以下の形式で出力する：

```
## レビュー結果: [ファイル名]

### 問題あり
- **[観点]** [該当箇所の引用] → [改善案]

### 軽微な指摘
- **[観点]** [該当箇所の引用] → [改善案]

### 良い点
- [具体的に良かった点]

### 総評
[全体的な評価と優先度の高い改善点のまとめ]
```

問題がなかった観点は省略してよい。改善案は具体的な修正文を提示する。
